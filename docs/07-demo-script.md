# 答辩演示脚本

## 演示目标

用 5 到 8 分钟讲清楚系统做什么、怎么流转、前后端如何配合、数据如何变化。

## 开场介绍

可以这样说：

「我们的项目是一个海洋环境巡检课程设计系统，采用前后端分离架构。后端使用 Laravel，前端使用 React + Vite，数据库默认使用 SQLite。系统围绕一条简化业务链展开：巡检任务、样本登记、检测结果录入、异常上报处理、简单分析建议和首页统计。」

## 演示步骤

### 1. 首页统计

打开首页，说明：

- 首页展示任务总数、样本数、异常数和分析建议。
- 首页是系统总览，适合管理人员快速了解巡检情况。
- 页面上可以看到完整业务链。

讲法示例：

「这里的统计数字来自后端 `/api/dashboard` 接口，后端从任务、样本、异常和分析表中汇总数据。」

### 2. 巡检任务

进入巡检任务页面。

操作：

1. 创建一个任务，例如「近岸海域水质巡检」。
2. 选择巡检区域和负责人。
3. 点击开始任务。

讲法示例：

「巡检任务是业务起点，后续样本都要归属于某个任务。任务状态从待开始变为进行中，最后可以提交完成。」

### 3. 样本登记

进入样本管理页面。

操作：

1. 选择刚才的任务。
2. 填写样本编号，例如 `S-20260612-001`。
3. 填写采样点和采样时间。
4. 保存样本。

讲法示例：

「样本表通过 `inspection_task_id` 关联巡检任务，所以一个任务下可以登记多个样本。」

### 4. 检测结果录入

进入样本详情或检测结果页面。

操作：

1. 为样本录入 pH、盐度、温度、溶解氧、浊度。
2. 可以故意输入一个异常值，例如 pH 为 `8.9` 或溶解氧为 `4.6`。
3. 保存检测结果。

讲法示例：

「检测结果是系统判断异常的依据。课程设计版本使用简单阈值规则，而不是复杂算法，这样逻辑更清楚，也便于答辩说明。」

### 5. 异常上报和处理

进入异常与分析页面。

操作：

1. 查看刚才生成或上报的异常。
2. 说明异常等级、异常指标和状态。
3. 填写处理说明。
4. 点击处理完成。

讲法示例：

「异常状态从待处理变为已处理，处理说明会保存到异常表中。这样系统不仅记录问题，也记录了处理结果。」

### 6. 简单分析建议

对样本点击分析。

讲法示例：

「分析建议根据检测指标和异常记录生成，例如提示复检、增加采样点或关注某个区域。这里不使用外部 AI 服务，避免课程设计过重。」

### 7. 返回首页

回到首页统计。

讲法示例：

「完成一轮业务操作后，首页统计会更新。这样就形成了从任务创建到结果统计的闭环。」

## 技术讲解顺序

建议按下面顺序回答老师问题：

1. **架构：** 前端 React，后端 Laravel，数据库 SQLite。
2. **数据：** 任务、样本、检测结果、异常、分析建议之间有关联。
3. **接口：** 前端通过 `/api/*` 调用后端 JSON 接口。
4. **规则：** 异常和分析建议通过简单阈值生成。
5. **简化：** 为课程设计去掉队列、完整权限、图片识别等复杂功能。

## 可能被问到的问题

### 为什么用 SQLite？

因为课程演示更关注业务流程和系统设计，SQLite 不需要单独安装数据库服务，便于团队成员和老师本地运行。

### 为什么不做完整权限？

完整 RBAC 会增加很多表和页面，但和本次核心业务链关系不大。课程设计版本只保留必要用户概念，把重点放在巡检业务流程上。

### 分析建议是不是 AI？

当前版本不是 AI 模型，而是规则分析。这样实现简单、结果可解释，适合答辩。如果后续扩展，可以接入模型或更复杂算法。

### 原项目和本项目是什么关系？

本项目是课程设计简化版，只参考技术方向和业务主题，不复制原项目的企业复杂度，也不修改原项目。
