# 系统设计

## 总体架构

本项目采用前后端分离架构。

```text
浏览器
  |
  | HTTP / JSON
  v
React + Vite 前端
  |
  | /api/*
  v
Laravel 后端
  |
  | Eloquent ORM
  v
SQLite 数据库
```

## 模块划分

| 模块 | 职责 |
| --- | --- |
| 前端页面 | 展示统计、列表、表单和操作按钮 |
| API 路由 | 定义前后端交互入口，例如 `/api/tasks` |
| Controller | 接收请求、校验参数、调用模型、返回 JSON |
| Model | 表示业务实体，例如任务、样本、检测结果 |
| Migration | 定义数据库表结构 |
| Seeder | 提供演示数据，方便答辩展示 |
| SQLite | 保存本地演示数据 |

## 业务状态设计

### 巡检任务状态

| 状态 | 含义 |
| --- | --- |
| pending | 待开始 |
| in_progress | 进行中 |
| submitted | 已提交 |

### 异常状态

| 状态 | 含义 |
| --- | --- |
| open | 待处理 |
| resolved | 已处理 |

### 分析状态

| 状态 | 含义 |
| --- | --- |
| completed | 已生成建议 |
| failed | 生成失败，通常用于预留扩展 |

## 数据流

### 创建任务

1. 用户在前端填写任务表单。
2. 前端调用 `POST /api/tasks`。
3. Laravel 创建 `inspection_tasks` 记录。
4. 前端刷新任务列表和首页统计。

### 登记样本和录入结果

1. 用户选择某个巡检任务。
2. 前端调用 `POST /api/samples` 创建样本。
3. 用户在样本详情页录入检测结果。
4. 前端调用 `POST /api/samples/{id}/results`。
5. 后端保存结果，并根据阈值判断是否需要创建异常。

### 异常处理和分析建议

1. 用户在异常列表中查看待处理异常。
2. 用户填写处理说明，调用 `POST /api/exceptions/{id}/resolve`。
3. 用户对样本发起分析，调用 `POST /api/samples/{id}/analyze`。
4. 后端根据检测值和异常记录生成文字建议。
5. 首页展示统计变化和最近建议。

## 简化分析规则

答辩时可说明：本系统不训练 AI 模型，而是使用规则判断，适合课程设计。

示例规则：

- pH 小于 6.5 或大于 8.5：提示水体酸碱度异常，建议复检。
- 溶解氧低于 5 mg/L：提示水体含氧量偏低，建议关注生态风险。
- 浊度过高：提示可能存在污染或泥沙扰动，建议增加采样点。
- 多项指标异常：建议提升异常等级并安排人工复核。

## 为什么这样设计

- **Laravel：** 路由、Controller、Model、Migration 结构清晰，适合 PHP 课程设计。
- **React/Vite：** 启动快，适合做前后端分离页面。
- **SQLite：** 不需要单独安装 MySQL，演示环境更稳定。
- **规则分析：** 逻辑直观，答辩时能讲明白。
- **少量表结构：** 覆盖完整业务链，同时避免系统过重。
